¿Por qué es importante diseñar antes de programar?

Imagina construir una casa sin planos. Podrías terminar con puertas que no abren o habitaciones sin salida. Lo mismo pasa con el software.

1
Ahorra tiempo — Corregir un boceto toma minutos; corregir código toma horas.
2
Valida con el cliente — El cliente ve cómo quedará antes de que exista.
3
Guía al equipo — Todos saben qué construir y cómo debe verse.

Paso 1 — Identificar el tipo de aplicación

Antes de dibujar cualquier pantalla, debemos saber qué estamos construyendo.

💻 Aplicación Independiente (Desktop)

Se instala en el computador. Funciona sin internet.

Ejemplo: Paint, Word, calculadora.
🌐 Aplicación Web

Funciona desde el navegador. Requiere servidor.

Ejemplo: Gmail, Moodle, tiendas en línea.

💡 El tipo de aplicación define las tecnologías, el diseño y la forma en que el usuario va a interactuar.

Paso 2 — Conocer la imagen institucional

La interfaz debe reflejar la identidad visual del cliente.

🎨 Colores corporativos
Usar la paleta de la empresa, no colores aleatorios.
🔤 Tipografía
Fuentes acordadas: tamaños, pesos y estilos.
🖼️ Logo y elementos
Posición correcta del logo, íconos y marca.
  • Solicitar el manual de identidad visual del cliente.
  • No inventar colores o tipografías propias sin autorización.
  • La consistencia visual genera confianza en el usuario.

¿Qué es un Mapa de Navegación?

Es un diagrama que muestra todas las pantallas de la aplicación y cómo el usuario puede moverse entre ellas.

🏠 Inicio
📋 Registro
🔑 Login
ℹ️ Acerca de
📊 Dashboard
👤 Perfil
🚪 Cerrar sesión

Cada caja = una pantalla | Cada flecha = una acción del usuario

¿Cómo construir el mapa de navegación?

1
Lista todas las pantallas — Inicio, Login, Registro, Dashboard, etc.
2
Define el punto de entrada — ¿Desde dónde empieza el usuario? (Splash screen o Login).
3
Conecta las pantallas — ¿Qué botón o acción lleva de una a otra?
4
Identifica flujos alternativos — Error de login, usuario sin permisos, 404.
5
Valida con el cliente — ¿Tiene sentido para quien lo va a usar?

Reglas de Usabilidad y Accesibilidad

👁️ Visibilidad
El usuario siempre sabe dónde está dentro de la app.
🎯 Consistencia
Los botones y menús se comportan siempre igual.
⚡ Eficiencia
El usuario logra su objetivo en pocos pasos.
♿ Accesibilidad
La app puede usarla alguien con discapacidad visual o motriz.
  • Usar alto contraste de colores (mínimo 4.5:1 según WCAG).
  • Texto alternativo en imágenes (alt="").
  • Navegación posible solo con teclado.

El Estándar IEEE 830

Es una norma internacional para documentar correctamente los requisitos de software.

Piensa en IEEE 830 como el "contrato" entre el desarrollador y el cliente: define exactamente qué debe hacer el sistema, cómo debe hacerlo y bajo qué condiciones.

  • Evita malentendidos entre cliente y desarrollador.
  • Sirve como base para hacer pruebas.
  • Permite estimar tiempos y costos con más precisión.

Las 6 secciones que debes incluir

# Sección ¿Qué describe?
01 Perspectiva del producto ¿Para qué existe esta aplicación? Contexto general.
02 Funciones del producto Lista de todo lo que hace la aplicación.
03 Características de los usuarios ¿Quién la usa? Experiencia, edad, conocimiento técnico.
04 Restricciones Limitaciones: hardware, plataforma, presupuesto, normativas.
05 Requisitos funcionales Acciones específicas que el sistema debe realizar.
06 Requisitos no funcionales Calidad: rendimiento, seguridad, disponibilidad, usabilidad.

Requisitos Funcionales vs No Funcionales

⚙️ Funcionales (RF)

Son las acciones que hace el sistema.

✅ El sistema permite registrar un usuario.
✅ El sistema muestra el historial de compras.
✅ El administrador puede eliminar productos.
🌟 No Funcionales (RNF)

Son las características de cómo lo hace.

🚀 Tiempo de respuesta menor a 2 segundos.
🔒 Las contraseñas deben cifrarse con SHA-256.
📱 La app debe funcionar en móvil y escritorio.

💡 Un RF dice "QUÉ" hace el sistema. Un RNF dice "CÓMO" lo hace.

Ejemplo: Sistema de Biblioteca

Sección Ejemplo
Perspectiva Sistema web para gestionar préstamos de libros en una biblioteca escolar.
Funciones Registrar libros, gestionar préstamos, generar reportes, buscar por categoría.
Usuarios Bibliotecario (adulto, manejo básico de PC) y Estudiante (joven, acceso móvil).
Restricciones Solo navegadores modernos. Servidor con 2GB RAM. Sin acceso a internet externo.
RF RF-01: El sistema permitirá registrar nuevos libros con ISBN, título y autor.
RNF RNF-01: La búsqueda de libros debe devolver resultados en menos de 1 segundo.

Proceso de Validación y Retroalimentación

Un prototipo no está listo hasta que pasa por revisión.

1
Crear el prototipo — Diseño inicial basado en los requerimientos.
2
Presentar al cliente — Mostrar el flujo completo, pantalla por pantalla.
3
Recoger retroalimentación — Anotar cambios, mejoras y errores detectados.
4
Ajustar el prototipo — Aplicar los cambios y versionar el documento.
5
Aprobar y firmar — El cliente firma el prototipo aprobado.

Herramientas para crear prototipos

🖊️ Papel y lápiz
Rápido. Para las primeras ideas.
📊 Balsamiq
Wireframes de baja fidelidad. Muy intuitivo.
🎨 Figma
Prototipos profesionales. Colaborativo.
🅰️ Adobe XD
Alta fidelidad. Exporta assets.
🖥️ Draw.io
Mapas de navegación y diagramas. Gratuito.
📝 Word / LibreOffice
Para el documento PDF final de la evidencia.

¿Qué debe tener el documento final?

El producto a entregar es un PDF con los prototipos y su documentación.

  • Portada: nombre del aprendiz, programa, fecha, código de evidencia.
  • Descripción general: tipo de app, imagen institucional.
  • Mapa de navegación: diagrama con todas las pantallas.
  • Prototipos: imagen de cada pantalla con descripción.
  • Sección IEEE 830: las 6 secciones para cada prototipo.
  • Tabla de requisitos: cada requisito vinculado a su pantalla.
  • Conclusiones y proceso de validación.

Tips para que tu evidencia sea excelente 🏆

  • Usa herramientas gratuitas como Figma o Draw.io. Son las que usa la industria real.
  • Cada pantalla del prototipo debe tener su RF correspondiente citado.
  • No mezcles la imagen institucional del cliente con tu propia creatividad.
  • Versiona tus prototipos: v1.0, v1.1, v2.0 después de cada retroalimentación.
  • Cuida la ortografía y la presentación del documento PDF.
  • El mapa de navegación debe cubrir todos los flujos, incluyendo errores.
  • Pide retroalimentación a un compañero antes de entregar (revisión entre pares).
