¿por qué?

¿Qué es el control de versiones?

Es un sistema que guarda el historial de cambios de tu código. Te deja volver atrás, ver quién cambió qué y trabajar en equipo sin pisarse.

  • Vuelves a cualquier versión anterior
  • Sabes qué, quién y cuándo cambió
  • Varias personas, un mismo proyecto
Git en una frase

Git toma "fotos" (commits) de tu proyecto. Tú decides cuándo, y siempre puedes regresar a una foto anterior.

Analogía: como guardar partidas en un videojuego. Cada commit es una partida guardada a la que puedes volver.
local vs remoto

Dos lugares, un mismo proyecto

Repositorio local

En tu computador

Vive en tu máquina. Aquí trabajas, haces commits y guardas tu historial aunque no tengas internet.

Repositorio remoto

En la nube (GitHub / GitLab)

Una copia en internet. Sirve de respaldo y es el punto donde el equipo se sincroniza.

La evidencia EV04 pide instalar y configurar ambos: tu Git local y el repositorio remoto.

paso 1

Instalar Git

Windows

Descarga el instalador desde git-scm.com y sigue el asistente. Incluye Git Bash.

Linux (Debian/Ubuntu)

Instálalo desde la terminal con el gestor de paquetes apt.

macOS

Con Homebrew o instalando las Command Line Tools.

bash — instalación
# Linux Debian / Ubuntu sudo apt update sudo apt install git # macOS con Homebrew brew install git
En Windows, recuérdales que el instalador trae "Git Bash", la terminal donde ejecutarán todos los comandos.
paso 2

Verificar y configurar

Comprueba que quedó instalado y dile a Git quién eres. Esos datos firman cada commit.

bash — configuración global
# ¿Quedó instalado? git --version git version 2.43.0 # Tu identidad (firma tus commits) git config --global user.name "Tu Nombre" git config --global user.email "tucorreo@ejemplo.com" # Revisar la configuración git config --list

--global aplica esa configuración a todos tus proyectos del equipo. Sin ella, los commits no tienen autor.

paso 3

Crear tu repositorio local

Dentro de la carpeta de tu proyecto, inicializas Git y haces tu primer commit.

bash — repositorio local
cd mi-proyecto git init # crea el repositorio local git status # ver el estado de los archivos git add . # prepara TODOS los cambios git commit -m "Primer commit" # guarda la "foto"

Este diagrama es la columna vertebral de toda la sección. Si los estudiantes entienden estos cuatro estados, el resto son detalles.
paso 1 del ejercicio

Crear el remoto y conectarlo

  1. Crea un repositorio público en GitHub llamado Programa-Git
  2. Añádelo a tu repositorio local con git remote add
  3. Lista los remotos configurados con git remote -v
bash — conectar remoto
# copia la URL https del repo creado en GitHub git remote add github https://github.com/usuario/Programa-Git.git git remote -v github https://github.com/usuario/Programa-Git.git (fetch) github https://github.com/usuario/Programa-Git.git (push)

github es solo el apodo del remoto. Lo usarás luego en el push.

paso 2 del ejercicio

Subir los cambios al remoto

Envías tus commits locales a GitHub y compruebas el historial de versiones en la web.

bash — push
git push github master Enumerating objects... done. To https://github.com/usuario/Programa-Git.git * [new branch] master -> master

Luego entra a GitHub y revisa la pestaña de commits: ahí verás el historial de versiones que pide la evidencia.

Hoy muchos repos usan la rama main en vez de master. El ejercicio usa master; usa el nombre que muestre tu repositorio.
paso 3 del ejercicio

El ejercicio kotlin.txt

  1. Crea kotlin.txt con tu nombre y tu correo
  2. Agrega los cambios y haz commit con el mensaje "Añadido datos"
  3. Sube los cambios al repositorio remoto
bash — kotlin.txt
echo "Nombre Apellido - correo@ejemplo.com" > kotlin.txt git add . git commit -m "Añadido datos." 1 file changed, 1 insertion(+) git push origin master

legibilidad

Estándares de nombramiento

Nombres claros = código que cualquiera entiende. En Java se usan estas convenciones: 

// Paquetes: minúsculas, sin guiones
package co.edu.sena.proyecto.dao;

// Clases: PascalCase (primera letra mayúscula)
public class UsuarioDAO {

    // Variables y atributos: camelCase
    private String nombreCompleto;
    private int totalRegistros;

    // Métodos: camelCase, empiezan con verbo
    public void registrarUsuario() { ... }
    public Usuario consultarPorId(int id) { ... }
}
Refuerza la lógica: clases nombran cosas (sustantivos), métodos hacen cosas (empiezan con verbo).
conexión

Conectarse a la base de datos con JDBC

JDBC es el puente entre tu código Java y la base de datos. Cuatro pasos: cargar driver, abrir conexión, ejecutar, cerrar. 

String url  = "jdbc:mysql://localhost:3306/mi_basededatos";
String user = "root";
String pass = "1234";

// Abrir conexión (try-with-resources la cierra sola)
try (Connection con = DriverManager.getConnection(url, user, pass)) {
    System.out.println("Conexión exitosa");
} catch (SQLException e) {
    System.out.println("Error: " + e.getMessage());
}

Recuerda agregar el conector JDBC (el .jar de MySQL) a las dependencias de tu proyecto.

La evidencia exige las cuatro funcionalidades funcionando en el módulo.

create + read

Insertar y consultar

// CREATE — usa PreparedStatement para evitar inyección SQL
String sql = "INSERT INTO usuario (nombre, correo) VALUES (?, ?)";
PreparedStatement ps = con.prepareStatement(sql);
ps.setString(1, "Ana");
ps.setString(2, "ana@correo.com");
ps.executeUpdate();

// READ — recorrer los resultados
ResultSet rs = con.createStatement()
                  .executeQuery("SELECT * FROM usuario");
while (rs.next()) {
    System.out.println(rs.getInt("id") + " - " + rs.getString("nombre"));
}
Explica por qué PreparedStatement con "?" es más seguro que concatenar strings: previene inyección SQL.
update + delete

Actualizar y eliminar

// UPDATE
String sql = "UPDATE usuario SET correo = ? WHERE id = ?";
PreparedStatement ps = con.prepareStatement(sql);
ps.setString(1, "nuevo@correo.com");
ps.setInt(2, 1);
ps.executeUpdate();

// DELETE
String del = "DELETE FROM usuario WHERE id = ?";
PreparedStatement pd = con.prepareStatement(del);
pd.setInt(1, 1);
pd.executeUpdate();

Patrón repetido: preparar la consulta → asignar parámetros → ejecutar. Domínalo una vez y lo aplicas a todo.
