¿Qué es el control de versiones?

Concepto

Versionamiento = la máquina del tiempo del código

Un sistema de control de versiones (VCS) registra cada cambio que haces en los archivos de un proyecto a lo largo del tiempo.

  • Guarda quién cambió qué, cuándo y por qué.
  • Permite volver atrás a cualquier versión anterior.
  • Deja que varias personas trabajen en paralelo y luego unan su trabajo.
Sin control de versiones

proyecto_final.zip
proyecto_final_v2.zip
proyecto_final_ahora_si.zip
proyecto_final_ahora_si_DEF.zip

…y nadie sabe cuál es la buena.

Por qué importa

Qué problemas resuelve

Problema cotidiano Con control de versiones
“Borré algo y ya no puedo recuperarlo” Recuperas cualquier versión anterior.
“Dañé el código que funcionaba” Vuelves al último estado estable.
“Pisamos el trabajo del otro” Cada quien trabaja en su rama y luego se fusiona.
“No sé quién hizo este cambio” Cada cambio queda firmado con autor y fecha.
“Se perdió el proyecto con el computador” Hay una copia respaldada en el servidor remoto.

Tipos de VCS

Centralizado vs. distribuido

Centralizado (ej. SVN)
  • Un único servidor guarda todo el historial.
  • Necesitas conexión para casi todo.
  • Si cae el servidor, el equipo se detiene.
Distribuido (ej. Git)
  • Cada quien tiene una copia completa del historial.
  • Trabajas sin internet y sincronizas después.
  • Más rápido y más resistente a fallos.

Trabajaremos con Git, el sistema distribuido más usado del mundo.

La herramienta

Git, en una frase

Una herramienta que toma fotografías (commits) de tu proyecto y las encadena en una línea de tiempo que puedes recorrer.

commit

Una foto guardada de tus cambios, con un mensaje.

branch / rama

Una línea de trabajo paralela para no afectar lo estable.

merge

Unir el trabajo de dos ramas en una sola.

Concepto clave de la EV03

Git local vs Git remoto

● Git local

El repositorio que vive en tu computador.

  • Trabajas y guardas commits sin internet.
  • Solo tú lo ves.
  • Es tu espacio de trabajo personal.
● Git remoto

El repositorio que vive en un servidor (GitHub, GitLab…).

  • Lo comparte todo el equipo.
  • Requiere conexión para sincronizar.
  • Es el respaldo y el punto de encuentro.

EV03 · Comandos

● Comandos básicos — Git local

Comando Qué hace
git init Crea un repositorio nuevo en la carpeta actual.
git status Muestra qué archivos cambiaron o están sin guardar.
git add <archivo> Agrega cambios al área de preparación (staging).
git commit -m "..." Confirma los cambios como una nueva versión.
git log Muestra el historial de commits.
git branch Lista, crea o elimina ramas.
git merge <rama> Fusiona otra rama con la actual.

EV03 · Comandos

● Comandos básicos — Git remoto

Comando Qué hace
git clone <url> Descarga una copia completa de un repositorio remoto.
git remote add origin <url> Conecta tu repo local con uno remoto.
git remote -v Lista los repositorios remotos conectados.
git push Sube tus commits locales al remoto.
git fetch Descarga cambios del remoto sin fusionarlos aún.
git pull Descarga y fusiona los cambios del remoto.

Estándares de codificación

EV02 · Concepto

Un acuerdo para escribir igual

Un estándar de codificación es un conjunto de reglas sobre cómo escribir el código: nombres, espacios, mayúsculas, estructura.

  • El código se lee mucho más de lo que se escribe.
  • Si todos escriben parecido, cualquiera entiende el código de otro.
  • Reduce errores y facilita el mantenimiento.

Mismo código, dos estilos: 

int x=obtenerd(a,b);   // ¿qué es?
int distancia = obtenerDistancia(origen, destino);

EV02 · Nombramiento

Variables y constantes

  • Variables en camelCase, con nombres que digan qué guardan.
  • Constantes en MAYUSCULAS_CON_GUION.
  • Nada de a, x1, dato2. 
// Variables → camelCase
int numeroEstudiantes = 30;
double promedioNotas = 4.2;
String nombreCompleto = "Ana López";

// Constantes → MAYUSCULAS
final double VALOR_MATRICULA = 1500000;
final int EDAD_MINIMA = 18;

EV02 · Nombramiento

Declaración de clases

  • Clases en PascalCase (cada palabra inicia en mayúscula).
  • Nombres en sustantivo: una clase representa una cosa.
  • Un archivo por clase, con el mismo nombre.
// Clases → PascalCase, sustantivo
public class Estudiante {
    private String nombre;
    private int edad;
}

public class GestorMatriculas {
    // ...
}

EV02 · Nombramiento

Declaración de métodos

  • Métodos en camelCase y empezando con un verbo.
  • El nombre debe decir qué hace el método.
  • Una sola responsabilidad por método.
// Métodos → camelCase + verbo
public double calcularPromedio() { ... }

public boolean estaMatriculado(int id) { ... }

public void enviarNotificacion(String correo) { ... }

EV02 · Más allá de los nombres

Otras buenas prácticas

Indentación

Misma sangría en todo el equipo (2 o 4 espacios). Estructura visible.

Comentarios

Explica el por qué, no el qué obvio. El código ya dice qué hace.

Paquetes

En minúscula: com.empresa.proyecto.

¿Y la integración continua?

El versionamiento es la base de la integración continua (CI): cada vez que alguien sube código al remoto, el sistema lo prueba automáticamente.

1. Subesgit push
→
2. Se pruebabuild + tests
→
3. Se integralisto o falla

Detecta errores temprano y mantiene el proyecto siempre funcionando.
