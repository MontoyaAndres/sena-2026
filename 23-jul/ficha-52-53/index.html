Concepto

¿Qué significa probar una API?

No estamos probando botones ni colores. Estamos probando el contrato: si yo envío estos datos a esta dirección, el servidor debe devolverme exactamente esta respuesta.

La prueba pasa cuando la respuesta es la esperada, incluso cuando los datos están mal. 

# Yo envío
POST /api/aprendices
{
  "nombre": "Laura Méndez",
  "ficha": 3186652
}

# El servidor responde
201 Created
{
  "id": 17,
  "nombre": "Laura Méndez"
}
Herramienta

Postman

Un cliente que envía peticiones HTTP a su API sin necesidad de tener el frontend terminado. Reemplaza al navegador cuando el navegador no alcanza.

Paso 1

Descargar desde postman.com e instalar. Existe versión de escritorio y versión web.

Paso 2

Crear una Collection con el nombre del proyecto. Ahí viven todas sus peticiones.

Paso 3

Agregar un Request por cada endpoint. Guardarlo, no dejarlo en pestañas sueltas.

Una colección exportada (.json) vale más que veinte pantallazos. Expórtenla y anéxenla.
Anatomía de una petición

Cuatro cosas que definen cada prueba

01 · Método

Qué quiero hacer

GET POST PUT DELETE

02 · URL

A dónde lo envío

http://localhost:8080/api/aprendices/3

03 · Headers

Cómo lo envío

Content-Type: application/json
Authorization: Bearer <token>

04 · Body

Qué le mando

El JSON con los datos. Solo aplica en POST, PUT y PATCH.

Referencia rápida

Métodos y códigos de respuesta

GET Consultar. No modifica nada.
POST Crear un registro nuevo.
PUT Actualizar un registro existente.
DEL Eliminar un registro.
200 OK. Salió bien.
201 Created. Se creó el recurso.
400 Datos mal formados.
401 / 403 Sin credenciales o sin permiso.
404 La ruta o el registro no existe.
500 Se rompió el servidor. Revise la consola.
Criterio

Por cada endpoint, tres pruebas

Camino feliz

Datos válidos. Debe responder 200 o 201 con la información correcta.

Datos inválidos

Campo obligatorio vacío, tipo de dato equivocado, correo mal escrito. Debe responder 400, no 500.

No existe

Pedir el id 9999. Debe responder 404 con un mensaje claro, no una pantalla de error.

Un testing que solo muestra el camino feliz no es un testing: es una demostración.

Entregable

El archivo ENDPOINT

Es el mapa de su API. Quien lo lea debe poder consumirla sin abrir su código.

Método Ruta Descripción Body / Params Respuesta
GET /api/aprendices Lista todos los aprendices — 200 · array
GET /api/aprendices/{id} Consulta uno por id id (path) 200 / 404
POST /api/aprendices Registra un aprendiz {nombre, ficha} 201 / 400
DEL /api/aprendices/{id} Elimina un aprendiz id (path) 204 / 404
Incluyan la URL base del ambiente donde probaron: local, servidor de pruebas o despliegue.
Entregables

El video y el documento de pruebas

Video

  • Diga su nombre, ficha y número de evidencia al iniciar.
  • Muestre Postman en ejecución, no diapositivas del proceso.
  • Recorra cada endpoint: envíe, muestre la respuesta y el código de estado.
  • Incluya al menos un caso de error y explique por qué la respuesta es correcta.

Documento con pruebas

  • Un pantallazo por prueba, legible y sin recortar la respuesta.
  • Debajo de cada imagen: qué se probó, qué se esperaba y qué ocurrió.
  • Marque resultado: aprobada o fallida. Las fallidas también se documentan.
  • Cierre con una tabla resumen de pruebas ejecutadas.

De piezas sueltas a un sistema

Hasta ahora cada uno construyó partes: la base de datos, la API, algunas vistas. La Guía 8 pide algo distinto: que esas partes se hablen entre sí y funcionen como una sola aplicación web.

Integrar no es pegar archivos en una carpeta. Es que un módulo consuma lo que otro produce, con contratos claros y sin duplicar código.

Materiales de formación
  • Construcción aplicación web
  • Construir aplicativo según las características de la arquitectura Android
  • Tecnologías emergentes y disruptivas
Antes de codificar

Los insumos ya los tienen

Nada de esto se inventa en la semana de entrega. Salió del análisis y el diseño que ya hicieron.

Requisitos

Funcionales y no funcionales, con acta de aprobación.

Diseño

Diagrama de clases, E-R, componentes, paquetes, actividad, secuencia y arquitectura.

Comportamiento

Casos de uso o historias de usuario, y el mapa de navegación.

Si su diagrama de clases no coincide con su código, uno de los dos está mal. Actualicen el que corresponda.
Arquitectura

Ubique cada componente en su capa

PresentaciónVistas, formularios, navegación. Consume la API, no consulta la base de datos.
ServiciosControladores y endpoints REST. Reciben, validan la forma y responden.
NegocioReglas del dominio. Aquí vive la lógica que el proyecto realmente resuelve.
DatosRepositorios, DAO, conexión y modelo de base de datos.

Por cada capa deben poder responder: qué librerías y qué framework usaron, y por qué.

Cómo se codifica

Buenas prácticas exigidas

  • Divida cada módulo en componentes reutilizables.
  • Organice el código en paquetes con nombres entendibles.
  • Aplique los patrones de diseño según la arquitectura definida.
  • Implemente los mecanismos de seguridad que requiere la aplicación.
  • Trabaje siempre sobre un repositorio de control de versiones.
  • Haga pruebas unitarias por módulo y guarde los resultados.
  • Documente configuraciones de servidor y base de datos.
  • Documente los ambientes de desarrollo y de pruebas.
Commits pequeños y con mensaje claro. Un único commit llamado «proyecto final» no evidencia proceso.

Advertencia

Lo que más se devuelve

En el testing

Pantallazos ilegibles, ningún caso de error, endpoints que no coinciden con el código entregado y videos sin audio ni explicación.

En la integración

Módulos que funcionan por separado pero no juntos, credenciales quemadas en el código y proyectos que no compilan en otra máquina.

En la documentación

Manuales que describen lo que se pensó hacer y no lo que quedó hecho.

En la entrega

Comprimido mal nombrado, archivos faltantes o subida en el espacio equivocado.
