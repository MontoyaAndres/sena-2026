GET /conceptos/api

¿Qué es un servicio web?

Es un programa que no tiene pantalla. Vive en un servidor, espera peticiones por HTTP y devuelve datos, casi siempre en JSON.

La metáfora del restaurante

Tú (el cliente) no entras a la cocina. Le pides al mesero (la API), él lleva tu orden a la cocina (el servidor) y te trae el plato (la respuesta).

Por qué sirve

Una sola API puede alimentar la app web, la app móvil y un tablero de reportes. La lógica se escribe una vez.

Insistir: la API es un contrato. El cliente no sabe si por dentro hay PHP, Java o Node. Solo conoce la URL, el método y el formato.

GET /conceptos/metodos-http

Los verbos de HTTP

El método le dice al servidor qué intención tiene la petición.

Método Intención Ejemplo en tu proyecto
GET Consultar, no modifica nada GET /api/usuarios
POST Crear un recurso nuevo POST /api/registro
PUT Reemplazar un recurso existente PUT /api/usuarios/7
PATCH Modificar solo algunos campos PATCH /api/usuarios/7
DELETE Eliminar DELETE /api/usuarios/7

El login usa POST aunque "solo consulte": la contraseña nunca debe viajar en la URL.

GET /conceptos/codigos-estado

Los números importan

Devolver 200 con un mensaje de error es el error más común de la evidencia.

Código Significa Cuándo lo usas
200 OK Todo salió bien Login correcto
201 Created Se creó el recurso Usuario registrado
400 Bad Request Datos mal enviados Falta la contraseña
401 Unauthorized No autenticado Clave incorrecta
409 Conflict Choca con algo existente El usuario ya está registrado
500 Server Error Se rompió tu código Nunca a propósito
Regla mnemotécnica: 2xx tú lo hiciste bien; 4xx el cliente lo hizo mal; 5xx yo lo hice mal.

POST /api/registro

La regla de oro

"SENA*2026"
texto plano
bcrypt ──▶
$2a$10$N9qo8uLO...
hash irreversible

La contraseña nunca se guarda tal cual. Se guarda su hash. Para verificar el login no se descifra: se vuelve a calcular y se comparan los hashes.

Si en tu base de datos puedes leer la contraseña de un usuario, está mal hecho.

POST /ev01/codigo · 2 de 3

Registro

// POST /api/registro → crea un usuario nuevo
app.post('/api/registro', async (req, res) => {

  const { usuario, contrasena } = req.body;        // 1. Leer los datos

  if (!usuario || !contrasena)                      // 2. Validar
    return res.status(400).json({ ok: false, mensaje: 'Datos incompletos' });

  if (usuarios.some(u => u.usuario === usuario))    // 3. ¿Ya existe?
    return res.status(409).json({ ok: false, mensaje: 'El usuario ya existe' });

  const hash = await bcrypt.hash(contrasena, 10); // 4. Nunca texto plano
  usuarios.push({ usuario, hash });                 // 5. Guardar

  res.status(201).json({ ok: true, mensaje: 'Usuario registrado' });
});
POST /ev01/codigo · 3 de 3

Inicio de sesión

// POST /api/login → valida las credenciales
app.post('/api/login', async (req, res) => {

  const { usuario, contrasena } = req.body;

  const encontrado = usuarios.find(u => u.usuario === usuario);
  const valido = encontrado && await bcrypt.compare(contrasena, encontrado.hash);

  // Mensaje genérico: no revelamos si falló el usuario o la contraseña
  if (!valido)
    return res.status(401).json({ ok: false, mensaje: 'Error en la autenticación' });

  res.status(200).json({ ok: true, mensaje: 'Autenticación satisfactoria' });
});

app.listen(3000, () => console.log('API en http://localhost:3000'));
Preguntar al grupo: ¿por qué el mismo mensaje para usuario inexistente y clave errada? Respuesta: para no permitir enumerar usuarios.

@RestController
public class LoginController {

  @PostMapping("/api/login")
  public ResponseEntity<Respuesta> login(@RequestBody Credenciales c) {

    if (!servicio.autenticar(c))   // valida contra el hash

GET /ev02/postman

¿Por qué Postman?

Tu API no tiene interfaz. El navegador solo sabe hacer GET. Postman te deja enviar cualquier método, con cualquier cuerpo y cualquier cabecera, y ver la respuesta cruda.

1. Instalar

Descárgalo de postman.com. Crea una cuenta gratuita para sincronizar tu colección.

2. Crear colección

Una colección por evidencia. Dentro, una petición por endpoint.

3. Exportar

La colección se exporta como .json y se adjunta a la entrega.

POST /ev02/peticion

Cómo se arma una petición

  • Método: POST
  • URL: http://localhost:3000/api/login
  • Pestaña Body → raw → tipo JSON 
{
  "usuario": "aprendiz01",
  "contrasena": "SENA*2026"
}

Si eliges Body → form-data en vez de raw/JSON, tu servidor recibirá undefined. Es el 90% de los "no me funciona, profe".

});

// Verifica el contenido del mensaje

  pm.expect(pm.response.json().mensaje).to.eql("Autenticación satisfactoria");
