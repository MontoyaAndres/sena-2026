Agenda01
Antes de empezar

¿Por qué modelar antes de codificar?

Imagina construir una casa sin planos: paredes que no calzan, tuberías repetidas, puertas que no abren.

Una base de datos sin modelo es exactamente eso: datos duplicados, errores al actualizar, información que se pierde.

Modelar = pensar antes de teclear.

Un buen modelo te ahorra horas de bugs, refactors y café derramado a las 2 a.m.

Todo cabe en una tabla

El modelo relacional, propuesto por E. F. Codd en 1970, dice algo simple: la información del mundo real se puede representar como tablas conectadas entre sí.

Modelo Relacional03
Parte 1 · Modelo Relacional

Anatomía de una tabla

  • Tabla / Relación: el conjunto completo (ej. estudiante).
  • Atributo / Columna: una propiedad (nombre, email).
  • Tupla / Fila: un registro completo (un estudiante real).
  • Dominio: los valores válidos de un atributo (ej. email → texto con @).
id nombre edad
001 María 22
002 Carlos 25

↑ Esta fila es una tupla

Modelo Relacional04
Parte 1 · Modelo Relacional

Las llaves son la clave

PK Llave Primaria

Identifica de forma única cada fila. No se repite, nunca está vacía.

→ id_estudiante

FK Llave Foránea

Apunta a la PK de otra tabla. Crea las relaciones entre ellas.

→ id_curso en matricula

Sin llaves, una tabla es solo una hoja de cálculo. Con llaves, es una base de datos.

Modelo Relacional06
Parte 2 · Cardinalidad

Cardinalidad:
¿cuántos con cuántos?

Cuando dos tablas se relacionan, hay que responder una pregunta:

"¿Cuántos registros de A pueden estar relacionados con cuántos de B?"

Hay tres tipos de respuesta. Vamos uno por uno.

Cardinalidad07
Cardinalidad · Tipo 1

Uno a Uno (1:1)

PERSONA
tiene 1
1
CÉDULA

Cada persona tiene exactamente una cédula, y cada cédula pertenece a una sola persona.

Otros ejemplos:
  • Un usuario ↔ un perfil
  • Un país ↔ una capital
  • Un empleado ↔ un casillero asignado
Cardinalidad08
Cardinalidad · Tipo 2

Uno a Muchos (1:N)

CLIENTE
realiza 1
N
PEDIDO

Un cliente puede tener muchos pedidos, pero cada pedido pertenece a un solo cliente.

Implementación: la tabla "muchos" (PEDIDO) recibe la FK que apunta al "uno" (CLIENTE).

Es la más común.
  • Un autor → muchos libros
  • Un curso → muchos estudiantes (si solo eligen uno)
  • Una categoría → muchos productos
Cardinalidad09
Cardinalidad · Tipo 3

Muchos a Muchos (N:M)

ESTUDIANTE
cursa N
M
CURSO

Un estudiante puede tomar varios cursos, y cada curso tiene varios estudiantes.

Truco clave: en la práctica, una relación N:M se rompe en dos relaciones 1:N usando una tabla puente (o intermedia).

Cardinalidad10
Cardinalidad · Resumen

Tres formas, una pregunta

1:1

Uno a Uno

Cada A se conecta con un solo B. Poco frecuente.

1:N

Uno a Muchos

La más común. La FK va en el lado "muchos".

N:M

Muchos a Muchos

Necesita una tabla puente con dos FKs.

Pregúntate siempre: ¿uno de A puede tener varios de B? ¿Y al revés?

Cardinalidad11
Parte 3 · Normalización

Normalización:
poner la casa en orden

Es un proceso para organizar las tablas y eliminar tres enemigos silenciosos:

Redundancia

El mismo dato repetido en muchas filas.

Anomalías

Errores al insertar, actualizar o borrar.

Inconsistencias

El mismo dato escrito de formas distintas.

Lo haremos en tres pasos: 1FN → 2FN → 3FN. Cada uno se construye sobre el anterior.
Normalización12
Caso de estudio

Empezamos con una tabla desordenada

matricula (sin normalizar)
id_est nombre cursos nom_curso profesor
001 María Gómez C01, C02 SQL, Python Ana, Luis
002 Carlos Ruiz C01 SQL Ana
003 Laura Peña C02, C03 Python, UX Luis, Sofía

Problemas: celdas con varios valores, datos repetidos, difícil de actualizar.

Normalización13
Primera Forma Normal

1FN · Un valor por celda

Regla: cada celda contiene un único valor atómico. Nada de listas, comas ni datos amontonados.
✗ Antes
est cursos
001 SQL, Python
003 Python, UX
→
✓ Después (1FN)
est curso
001 SQL
001 Python
003 Python
003 UX

Cada fila representa una sola cosa. Cada columna, un solo dato.

Normalización14
Segunda Forma Normal

2FN · Sin dependencias parciales

Regla: estar en 1FN y que cada atributo no clave dependa de toda la llave primaria, no solo una parte.

Si tu PK es compuesta (ej. id_est + id_curso), ningún campo debe depender solo de una parte de ella.

✗ Mal
id_est id_curso nom_curso
001 C01 SQL
002 C01 SQL

nom_curso solo depende de id_curso.

→
✓ Bien (2FN)
id_est id_curso
001 C01
id_curso nom_curso
C01 SQL
Normalización15
Tercera Forma Normal

3FN · Sin dependencias transitivas

Regla: estar en 2FN y que ningún atributo no clave dependa de otro atributo no clave.
✗ Mal
id_curso profesor tel_prof
C01 Ana 3001234
C02 Luis 3015678

tel_prof depende de profesor, no del curso.

→
✓ Bien (3FN)
id_curso id_prof
C01 P1
id_prof nombre tel
P1 Ana 3001234

Si cambias el teléfono de Ana, lo haces en un solo lugar.

Normalización16
Normalización · Recap

Las 3 reglas en una frase

1FN

"Un valor por celda."

Sin listas ni datos múltiples en una columna.

2FN

"Depender de toda la PK."

Sin dependencias parciales en PK compuestas.

3FN

"Solo depender de la PK."

Sin dependencias entre atributos no clave.

Regla práctica: si un dato se repite muchas veces sin razón, probablemente quiere irse a otra tabla.
Normalización17
Para tu evidencia GA6-AA1-EV01

Checklist antes de entregar

  • Cada tabla tiene una llave primaria clara.
  • Las relaciones usan llaves foráneas bien identificadas.
  • La cardinalidad está marcada en cada relación (1:1, 1:N, N:M).
  • Las relaciones N:M usan tabla puente.
  • Cada celda tiene un único valor (1FN).
  • No hay dependencias parciales (2FN).
  • No hay dependencias transitivas (3FN).
  • El documento tiene portada, introducción, objetivo y análisis.
