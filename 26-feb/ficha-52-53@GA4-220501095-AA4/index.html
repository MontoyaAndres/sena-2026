Evaluación de artefactos del software

Objetivo de la sesión: entender qué son los artefactos, cuáles existen, cómo se verifican/validan y cómo construir instrumentos (listas de chequeo e informes) para evaluar su calidad.

1) ¿Qué es un artefacto?

Un artefacto de software es cualquier documento o producto que se crea durante el desarrollo: ayuda a planear, diseñar, construir, probar o mantener el sistema.
  • Puede ser un documento (requisitos), un diagrama, una historia de usuario…
  • También puede ser código, casos de prueba, manuales, reportes, etc.
  • La clave: sirve como evidencia de lo que se hizo y cómo se hizo.

¿Por qué se evalúan los artefactos?

Idea principal: detectar defectos lo más temprano posible cuesta menos tiempo y esfuerzo.
Beneficios:
  • Menos retrabajo
  • Más claridad en requisitos y diseño
  • Mejor comunicación del equipo
  • Mayor calidad del producto final

2) Tipos de artefactos (con ejemplos)

Gestión / planeación
  • Plan del proyecto
  • Cronograma
  • Matriz de riesgos
Requisitos
  • Historias de usuario
  • Casos de uso
  • Documento SRS
Diseño
  • Diagramas UML
  • Arquitectura
  • Modelo de datos
Construcción
  • Código fuente
  • Scripts (BD / despliegue)
  • Configuraciones
Pruebas
  • Casos de prueba
  • Plan de pruebas
  • Reporte de bugs
Entrega / operación
  • Manual de usuario
  • Guía de instalación
  • Notas de versión

3) Verificación vs Validación

Verificación

¿Estamos construyendo el producto correctamente?

  • Revisar si el artefacto cumple estándares, formato, reglas.
  • Ej.: lista de chequeo para un diagrama UML.
Validación

¿Estamos construyendo el producto correcto?

  • Confirmar si satisface la necesidad real del usuario/cliente.
  • Ej.: revisar requisitos con el usuario final.

Truco fácil: verificación = reglas/estándares; validación = necesidad/uso real.

4) ¿Qué es la evaluación de artefactos?

Es el proceso de revisar, medir y juzgar la calidad de los artefactos para saber si sirven para alcanzar las metas del proyecto.
  • Busca errores, inconsistencias, omisiones y oportunidades de mejora.
  • Se apoya en estándares (por ejemplo, guías internas, IEEE/ISO, buenas prácticas).
  • Termina con hallazgos, acciones y un informe.

5) ¿Cómo se realiza la evaluación?

  1. Definir alcance: qué artefactos se van a evaluar y para qué.
  2. Elegir criterios: completitud, claridad, consistencia, trazabilidad, formato, etc.
  3. Seleccionar instrumentos: listas de chequeo, revisiones, métricas, pruebas.
  4. Ejecutar la evaluación: revisar, registrar evidencias, marcar incidencias.
  5. Analizar resultados: severidad, causa, impacto, prioridad.
  6. Proponer mejoras: correcciones y responsables.
  7. Reportar: informe final con conclusiones y recomendaciones.

Criterios típicos para evaluar artefactos

Calidad del contenido
  • Completitud (¿falta algo?)
  • Claridad (¿se entiende?)
  • Consistencia (¿no se contradice?)
  • Trazabilidad (¿se conecta con requisitos?)
Calidad de forma
  • Formato y estructura
  • Ortografía y redacción
  • Uso correcto de plantillas
  • Versionado y control de cambios

6) ¿Qué instrumentos se utilizan?

Instrumentos “humanos”
  • Revisión por pares
  • Inspección (más formal)
  • Walkthrough (recorrido guiado)
  • Reuniones de validación con usuario
Instrumentos “documentales”
  • Listas de chequeo (checklists)
  • Plantillas y estándares
  • Matrices (trazabilidad, riesgos)
  • Métricas (defectos, cobertura, etc.)

En esta actividad, el instrumento estrella son las listas de chequeo y el informe.

7) ¿Qué resultados se obtienen?

  • Hallazgos: defectos, inconsistencias, faltantes.
  • Clasificación: severidad (alta/media/baja) y prioridad (urgente/normal).
  • Acciones: correcciones, mejoras, responsables y fechas.
  • Decisiones: aprobar, aprobar con cambios, rechazar y rehacer.
  • Lecciones aprendidas: para mejorar el proceso.
Meta: que el artefacto quede listo para avanzar a la siguiente fase sin arrastrar errores.

Esto luego se complementa con las listas de chequeo (EV03).

EV03 — Listas de chequeo para validación (documentación de diseño)

Producto Documento con listas de chequeo (PDF)

Una buena lista de chequeo tiene:
  • Nombre claro (¿para qué sirve?)
  • Categorías/variables (qué se espera encontrar)
  • Escala de incidencias (ej.: Cumple/No cumple/Parcial)
  • Cuadrícula clara y fácil de usar
  • Diseño básico (mientras más simple, mejor)

EV03 — Ejemplo de checklist (Diseño UML)

Lista de chequeo: Diagrama de clases
  • ¿Incluye clases principales del dominio?
  • ¿Relaciones (asociación/herencia) están correctas?
  • ¿Atributos y métodos tienen nombres claros?
  • ¿Multiplicidades están definidas?
  • ¿El diagrama es consistente con los requisitos?

Escala sugerida: Cumple / Parcial / No cumple / No aplica

Puedes hacer varias listas: arquitectura, BD, interfaces, etc.

EV04 — Informe de evaluación de artefactos de diseño

Debe incluir mínimo:
  • Resumen
  • Introducción
  • Objetivos

EV04 — Contenido clave del informe (qué mostrar)

  • Artefactos evaluados: lista + versión + fecha.
  • Criterios: completitud, claridad, consistencia, trazabilidad, formato.
  • Instrumentos usados: checklists, revisión por pares, registro de incidencias.
  • Hallazgos: tabla con incidencia, severidad, evidencia, recomendación.
  • Decisión: aprobado / aprobado con cambios / rechazado.
Consejo: usa una tabla simple de hallazgos (ID, Descripción, Severidad, Acción).

Errores comunes (y cómo evitarlos)

Errores típicos
  • Checklist demasiado larga
  • Criterios ambiguos (“bien hecho”)
  • No registrar evidencias
  • Sin conclusiones ni decisión final
Soluciones
  • Máximo 10–20 ítems por checklist
  • Ítems medibles (Sí/No/Parcial)
  • Observaciones concretas
  • Resumen + decisión + plan de mejora

Mini caso práctico (ejemplo)

Proyecto: App de citas médicas
Artefactos de diseño: diagrama de clases, diagrama de secuencia, modelo entidad-relación
Instrumentos: 3 checklists (uno por artefacto) + registro de incidencias
Resultado: “Aprobado con cambios” (se corrigen 5 inconsistencias antes de pasar a construcción)

Cierre

  • Los artefactos son todo lo que se produce en el proyecto (documentos, modelos, código…).
  • Evaluarlos temprano reduce costos y mejora calidad.
  • Las listas de chequeo y los informes hacen la evaluación clara y repetible.
  • EV01–EV04 piden: explicar, crear instrumentos, diseñar checklists y reportar resultados.
Si quieres, también puedo generarte las plantillas listas para copiar (portadas, tablas de hallazgos, checklists por tipo de documento) para que las exportes a PDF.
