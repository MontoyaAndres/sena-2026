Ficha 200 — Validar artefactos de análisis

Actividad GA2-220501093-AA3

Meta: verificar y validar los documentos de análisis (requisitos, modelos, especificaciones) para asegurar que están alineados con el informe de requisitos y lo que pidió el cliente.

Duración: 24 horas • Componente sugerido: “Validación de documentos”.

  • ¿Qué son artefactos de análisis?
  • ¿Qué significa validar y verificar documentos?

1) ¿Qué son “artefactos de análisis”?

Son los documentos/modelos que describen qué necesita el sistema y cómo se entiende el problema antes de diseñar y programar.
Requisitos
  • RF / RNF
  • Historias de usuario
  • Casos de uso
Modelos
  • Modelo de dominio
  • Procesos / BPMN
  • Casos de uso (UML)
Especificaciones
  • Glosario
  • Reglas de negocio
  • Prototipos (wireframes)

¿Por qué es tan importante validarlos?

Porque si el análisis está mal…
  • El diseño sale mal
  • El desarrollo se retrasa
  • Se construye algo que el cliente no pidió
Y si se valida a tiempo…
  • Menos cambios costosos
  • Más claridad
  • Mejor comunicación con el cliente

2) Verificación vs Validación (fácil)

Verificación

¿Está bien hecho el documento?

  • Formato, estructura, consistencia
  • Ortografía, claridad
  • Uso de plantilla y versión
Validación

¿Dice lo correcto según el cliente?

  • Requisitos completos
  • Lo entiende y aprueba el usuario
  • Coincide con el informe de requisitos

Truco: verificar = “calidad del documento”; validar = “calidad de la necesidad”.

3) Proceso simple para validar artefactos

  1. Identificar los documentos a validar (lista de artefactos).
  2. Definir criterios (qué debe cumplir cada documento).
  3. Preparar instrumentos (listas de chequeo + formato de hallazgos).
  4. Revisar documento por documento (marcar Cumple/No/Parcial).
  5. Registrar hallazgos (qué está mal y evidencia).
  6. Validar con usuarios/cliente (confirmar requisitos).
  7. Emitir decisión: Aprobado / Aprobado con cambios / Rechazado.

Evidencias de la actividad

EV01 y EV02 (qué son y cómo hacerlas)

EV01 — Mapa conceptual sobre validación

Producto Documento con mapa conceptual (PDF)

Objetivo: mostrar una secuencia lógica de dependencias para validar documentos del proyecto.
Debe incluir
  • Categorías y relaciones claras
  • Orden lógico (de lo general a lo específico)
  • Proporción y lectura fácil
  • Metodología de desarrollo (ágil, tradicional, etc.)
También mencionar
  • Documentos a validar (requisitos, modelos, etc.)
  • Formatos de aceptación (aprobado, con cambios, rechazado)
  • Herramienta usada (draw.io, Lucidchart, etc.)

EV01 — Idea de mapa conceptual (contenido sugerido)

Nodo central: “Validación de documentos de análisis”
  • Entradas: Informe de requisitos / necesidades del cliente
  • Artefactos: RF, RNF, casos de uso, reglas de negocio, glosario, prototipos
  • Criterios: completitud, claridad, consistencia, trazabilidad
  • Instrumentos: checklist, revisión por pares, reunión con usuario
  • Resultados: hallazgos + acciones + aceptación (aprobado / con cambios / rechazado)

Tip: usa flechas con verbos: “se valida con”, “se verifica mediante”, “genera”.

EV02 — Informe de análisis + listas de chequeo

Producto Informe técnico (PDF) — extensión libre

Propósito: validar que los artefactos de análisis están acordes con lo requerido por el cliente y que el cliente/usuarios los aprueban.
El documento debe tener mínimo
  • Portada
  • Introducción
  • Alcance
  • Lista de requerimientos
  • Versión del documento
Además debe evidenciar
  • Cumplimiento paso a paso
  • Estrategias de cumplimiento
  • Procesos y usuarios interesados
  • Validación con usuarios
  • Listas de chequeo paso a paso

EV02 — Estructura sugerida del informe

  1. Portada (evidencia, nombre, ficha, fecha)
  2. Control de versiones (v1.0, v1.1…)
  3. Introducción (qué se va a validar y por qué)
  4. Alcance (qué incluye y qué no incluye)
  5. Requerimientos (lista RF y RNF)
  6. Artefactos a validar (tabla: nombre, versión, responsable)
  7. Metodología (cómo se hizo la validación + instrumentos)
  8. Checklists (por artefacto)
  9. Hallazgos (tabla + severidad)
  10. Conclusiones + Aceptación (firmas o visto bueno)

EV02 — Ejemplo de checklist (Requisitos funcionales)

Checklist: RF (Escala: Cumple / Parcial / No cumple / N/A)
  • ¿Cada requisito tiene un ID único (RF-01, RF-02…)?
  • ¿Está redactado de forma clara y sin ambigüedad?
  • ¿Describe una sola necesidad (no mezcla varias)?
  • ¿Tiene criterios de aceptación (cómo saber si cumple)?
  • ¿Es trazable al objetivo del cliente (evidencia)?

Tip: máximo 10–15 ítems por checklist para que sea usable.

EV02 — Tabla simple de hallazgos (recomendado)

Campos sugeridos:
  • ID hallazgo (H-01, H-02…)
  • Artefacto (Requisitos / caso de uso / prototipo…)
  • Descripción (qué está mal o qué falta)
  • Severidad (Alta / Media / Baja)
  • Recomendación (qué corregir)
  • Responsable + fecha

Resultado final: Aprobado / Aprobado con cambios / Rechazado.

Tips para que te quede excelente (y fácil)

Mapa conceptual (EV01)
  • Usa 6–10 nodos clave
  • Flechas con verbos (“se valida con”)
  • Orden: Requisitos → Artefactos → Instrumentos → Resultados
Informe + checklists (EV02)
  • Una checklist por artefacto
  • Ítems medibles (no “está bonito”)
  • Hallazgos con evidencia

Cierre

  • Validar artefactos de análisis evita construir “lo incorrecto”.
  • EV01: mapa conceptual con relaciones lógicas y formatos de aceptación.
  • EV02: informe con alcance, requisitos, versión y checklists paso a paso.
